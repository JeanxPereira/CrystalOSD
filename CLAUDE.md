# CrystalOSD — PS2 OSDSYS Reconstruction

> Clean-room reconstruction of the PlayStation 2 OSDSYS (system menu), built by analyzing the binary in Ghidra and cross-referencing against PS2SDK and PCSX2 source.

## Architecture
- **Target**: HDDOSD/HOSDSYS 1.10U (OSDSYS_A.XLF from SUDC4, MIPS R5900)
- **Toolchain**: ps2dev binutils 2.45 + gcc 15.2 (extracted to `~/ps2dev/`)
- **Build**: splat split → asm assemble → ld link → byte-perfect ELF
- **Output**: Native PS2 ELF, testable on PCSX2
- **Future**: Desktop port via SDL2+OpenGL (shared source tree)

## Binary Stats
- Binary: `OSDSYS_A_XLF_decrypted_unpacked.elf` (3.7 MB, base `0x200000`, single LOAD seg)
- Identical to `hddosd.elf` (HDDOSD 1.10U) — same 3,864,601 bytes
- Entry point: `0x200008` (after 8 bytes of nop padding)
- GP value: `0x377970`; bss size: `0x199de0`
- 1,675 functions imported into Ghidra (from community `symbol_addrs.txt`)
- 4,619 data labels imported; 6,372 total symbols
- Analysis: MIPS-R5900, Aggressive Instruction Finder OFF
- `main` at `0x0020d4d0` — standalone, 429 lines, 921 instructions

## Decomp Workflow
1. Identify target function in Ghidra (ghidra-mcp)
2. Decompile → analyze control flow and data structures
3. Cross-reference with PS2SDK (`/crossref`)
4. Cross-reference with PCSX2 source for hardware behavior
5. Write clean C reconstruction in `src/<subsystem>/`
6. Add Ghidra address comment above every reconstructed function
7. Iterate match on decomp.me (`tools/decomp_match.py`)
8. Build full ELF with `make elf` → test on PCSX2

## Batch Export (Ghidra → Stubs → Transmuter)
For bulk processing, use the batch pipeline instead of manual function-by-function decompilation:

1. **Export stubs from Ghidra**: Run `BatchExportToC.java` in Ghidra Script Manager
   - Select CrystalOSD project root when prompted
   - Reads `asm/` to find unmatched functions (skips `FUN_*`, `gap_*`, `j_*`)
   - Skips functions already decompiled in `src/` or `src/stubs/`
   - Resolves addresses from `symbol_addrs.txt`
   - Exports to `src/stubs/<subsystem>/<function>.c`
2. **Run Transmuter on stubs**: `./tools/run_batch_transmuter.sh [subsystem]`
   - No args = all subsystems; with arg = single subsystem (e.g., `graph`)
3. **Review and promote**: Move cleaned stubs from `src/stubs/` to `src/` after verification
4. **Match**: Run `tools/decomp_match.py` or `tools/decomp-permuter` on promoted files

> ⚠️ `src/stubs/` contains raw Ghidra output (`uVar1`, `puVar2`, etc.) — never commit directly to `src/`

## Build Guardrails — CRITICAL (read before touching build infra)

The build pipeline has several fragile parts that were painstakingly debugged. **DO NOT modify these without understanding why they exist:**

### configure.py Post-Processing Pipeline
`configure.py` does three things after running splat:
1. **`fix_linker_script()`** — Moves `module_storage` from `.main` to its own `.main_modstor` section at `0x50A780`. Without this, module_storage (2.6 MB) sits between `.data` and `.sdata`, pushing `.sdata`/`.sbss` far beyond ±32KB of `$gp` (0x377970), causing **GPREL16 relocation overflow**.
2. **`fix_linker_script()`** also replaces `ALIGN(., 16)` with `ALIGN(., 4)` — the original binary uses 4-byte alignment between data sections, not 16.
3. **`fix_bss_section_attrs()`** — Replaces `.float 0` with `.word 0x00000000` in bss/sbss assembly files. MIPS GAS hardcodes `.bss` as NOBITS and refuses `.float` directives, even with `@progbits`.

### Critical splat_config.yml Settings
```yaml
subalign: 4              # NOT 8 — original binary is 4-byte aligned
ld_bss_is_noload: False  # materializes bss as data (needed for byte-perfect match)
section_order: [".text", ".data", ".rodata", ".sdata", ".sbss", ".bss", ".module_storage"]
```

### Makefile Linker Flags
```makefile
LDFLAGS := -m elf32lr5900 -EL -nostdlib --no-check-sections -G 0 --defsym=_gp=0x377970 -e 0x200008 -s
```
- **`-G 0`** (NOT `-G 0x10000`): Prevents linker from placing data in small-data sections, avoiding GPREL16 overflow.
- The ELF header is replaced post-link: the Makefile strips the linker-generated header and prepends the original 4KB header from `OSDSYS_A_XLF_decrypted_unpacked.elf`.

### Known Solved Issues (DO NOT RE-INTRODUCE)
| Issue | Root Cause | Fix |
|-------|-----------|-----|
| `R_MIPS_GPREL16` relocation overflow | module_storage (2.6MB) placed inside `.main`, pushing sdata beyond GP reach | `configure.py` moves it to `.main_modstor` |
| `small-data section too large` | `-G 0x10000` + splat layout | Changed to `-G 0` |
| `.float 0` in `.bss` | GAS refuses floats in NOBITS sections | `configure.py` rewrites to `.word 0x00000000` |
| `ALIGN(., 16)` padding mismatch | splat generates 16-byte alignment, original uses 4 | `configure.py` patches to `ALIGN(., 4)` |
| `subalign: 8` padding | splat pads subsections to 8 bytes, original uses 4 | Changed to `subalign: 4` |
| Exit/_exit symbol conflicts | stdlib names collide with kernel syscalls | Renamed in splat_config to `Exit_kernel`/`_Exit_syscall` |

## decomp-permuter (Brute-force Matching)
Submodule at `tools/decomp-permuter/`. Randomly permutes C source to find matching compiler output.
Supports MIPS natively — ideal for "last mile" matching where code is close but exact syntax is elusive.

```bash
# Import a function for permutation
./tools/permuter_import.sh <func_name> <src_file>
# Example:
./tools/permuter_import.sh gsAllocBuffer src/graph/gs_util.c

# Run permuter (multi-threaded)
cd tools/decomp-permuter
python3 ./permuter.py nonmatchings/<func_name> -j8
```

PERM macros can be used in source to guide permutation:
- `PERM_GENERAL(a, b)` — try both alternatives
- `PERM_RANDOMIZE` — enable random permutation in a region
- `PERM_LINESWAP(lines)` — try all orderings of statements

Compile script: `tools/permuter/compile.sh` (wraps ee-gcc with project flags).
